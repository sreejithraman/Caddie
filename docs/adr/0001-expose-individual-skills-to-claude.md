# Expose individual canonical skills at each user harness root

Claude Code 2.1.204 did not discover skills when `.claude/skills` itself was a symlink to `.agents/skills`, but did discover the same canonical skill when `.claude/skills/<name>` was an individual directory symlink. Caddie therefore keeps each harness skills directory real and creates one Caddie-owned link per selected canonical skill; it does not maintain a second skill copy.

Project Skills are already canonical beneath the project `.agents/skills`, so only Claude needs a project-local link. User Skills are canonical beneath Caddie's configured User Skills scope rather than the operating-system home. Bootstrap, Adoption, and user reconciliation therefore create matching individual links under both the actual `~/.agents/skills` and `~/.claude/skills` roots. Plan effects may write outside the configured User Skills scope only at those two runtime-HOME roots, and validation repeats that restriction during apply and recovery. Caddie's narrowly fixed machine-local coordination exception is `~/.agents/.caddie/user-mutation.lock` plus `user-operation.json`; real-ancestor checks guard that directory, and these files serialize and reserve user-harness recovery rather than representing plan effects.

When Codex already has a real User Skill directory at its harness path, Caddie uses a distinct Adoption operation rather than broadening normal link creation. The approved plan binds the original fingerprint, copies it to an absent canonical destination, then replaces it through a journaled, crash-recoverable pair of renames; the path can be briefly absent between them. Recovery can restore the original directory byte-for-byte. Existing Claude links that point through the Codex path are left unchanged and unowned; direct links created by Caddie remain ledger-owned. A runtime-HOME lock serializes all Caddie user-harness mutations across canonical scope roots, and a preparing reservation is published before the scope journal so a crash cannot reopen cross-scope mutation until matching recovery completes.

The release gate runs the installed Claude binary with an absent-skill control and must fail when Claude is unavailable or an exposed skill is not discovered.
